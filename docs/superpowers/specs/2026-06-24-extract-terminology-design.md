# 用語・表記ルール抽出機能 設計書

## 背景・目的

KRAFTON Japanの過去ニュースリリース371件をPR TimesからNotion DB(ニュースリリースDB)に取り込み済み。これらは社内で実際に使われてきた公式な日本語表記のコーパスである。このコーパスから

1. ゲームタイトル・キャラクター・組織名などの**固有名詞の正式表記**(用語集DB向け)
2. 「タイトル名は『』で囲む」「ユーザーではなくプレイヤーと呼ぶ」といった**文体・表記スタイルのルール**(用語集DBのスキーマには収まらない)

を自動抽出し、ローカライズ作業(`localize_app.py`)の翻訳品質・表記統一に活用できる状態を作る。

将来的にはHQから届くKR/EN原文と日本語訳のペアからも同様に用語・ルールを抽出し、用語ノブレ(表記ゆれ)を継続的に解消していく構想がある。今回はその第一段(日本語原文コーパスのみからの抽出)を実装するが、抽出ロジックは日本語単独でもKR/EN原文付きでも呼び出せる汎用的な形にしておく。

## スコープ

- 対象データ: 既にNotionにインポート済みの371件のニュースリリース(日本語原文)
- 用語抽出と表記ルール抽出を同じバッチ処理の中で同時に行う
- 差分再実行に対応する(初回は371件全件、以降は新規追加分のみ処理できる)
- 実行方式: `prtimes_to_notion.py`と同様の単体Pythonスクリプト(Streamlit UIへの統合は今回のスコープ外)

## 全体アーキテクチャ

新規スクリプト `extract_terminology.py` を `prtimes_to_notion.py` と同じディレクトリに追加する。

```
ニュースリリースDBをクエリ（data_sources.query）
  → 「用語抽出済み」チェックボックスが未設定のreleaseのみ取得（差分実行）
  → 各releaseのPR Times URLからfetch_article_body()で本文を再取得
     （prtimes_to_notion.pyの既存関数をimportして再利用）
  → 取得できたreleaseを10件ずつバッチ化
  → 各バッチをClaude APIに1回投げ、用語候補＋表記ルール候補を同時抽出
      → 用語候補: 既存（pending＋approved）と重複しないものだけ即時pending登録
      → 表記ルール候補: メモリ上のリストに蓄積（根拠release URLを保持）
  → 全バッチ処理後、蓄積した表記ルール候補をClaude APIに1回渡して統合・重複排除
  → 統合済みルールをNotionページ「KRAFTON Japan 表記ルール」の「要確認」セクションに追記
     （既に同一文言のルールがあれば追記しない）
  → 処理した各releaseの「用語抽出済み」チェックボックスをON
```

## 詳細設計

### 本文取得

`prtimes_to_notion.fetch_article_body()` を import して再利用する。Notionに保存済みのページブロックを読み返すのではなく、DB上の「PR Times URL」プロパティから直接PR Timesへ再取得する方が、既存の実装をそのまま流用できてシンプルになる。取得失敗時は該当releaseをスキップしログ出力する（既存スクリプトと同方針）。

### Claude APIプロンプト設計

1バッチ（release 10件分の本文）につき1回のClaude呼び出しで、以下のJSON形式を1回で取得する（用語・表記ルールを別々に呼ぶとコストが倍になるため統合する）。

```json
{
  "terms": [
    {"ja": "日本語表記", "en": "英語表記（本文中に原語表記があれば。無ければ空文字）", "category": "タイトル名|キャラクター|ゲームシステム|イベント・大会|組織・役職|その他"}
  ],
  "style_rules": [
    {"rule": "ルールの説明", "example": "Before→After、または本文からの引用例"}
  ]
}
```

抽出関数は将来`localize_app.py`からKR/EN原文付きで呼び出せるよう、`extract_terms_and_style(text_ja: str, text_kr: str = "", text_en: str = "") -> dict` のような汎用シグネチャにする。今回は`text_kr`/`text_en`は渡さない（PR Times原文コーパスには存在しないため）。共有モジュール化は今回は行わず、`extract_terminology.py`内に定義する（将来必要になった時点でモジュール分割する）。

### 用語の重複排除・書き込み

スクリプト開始時に用語集DBの既存エントリ（pending＋approved）を全件取得し、「日本語表記」の集合をメモリに保持する。バッチ処理中に新しい用語候補が出るたびにこの集合と比較し、未知のものだけ

```python
notion.pages.create(
    parent={"database_id": NOTION_GLOSSARY_DB_ID},
    properties={
        "日本語表記": ...,
        "EN表記": ...（あれば）,
        "カテゴリ": ...,
        "ステータス": {"select": {"name": "pending"}},
        "抽出元リリース": {"url": 該当release URL},
    },
)
```

で追加する。追加した用語はその場で集合にも加え、同一バッチ内・以降のバッチでの重複も防ぐ。KR表記は今回のコーパスには存在しないため空欄のままになる。

### 表記ルールの統合・書き込み

各バッチの`style_rules`を、根拠となるrelease URL・タイトルとともにメモリ上のリストに蓄積する。全バッチ完了後、蓄積した候補（重複・類似表現が多いはず）をもう一度Claude APIに1回渡し、「類似・重複するルールを統合し、一意で明確な最終リストにしてください」と指示して統合済みリストを得る。

統合結果は、新規Notionページ「KRAFTON Japan 表記ルール」の「要確認」セクションに箇条書きで追記する（ルール内容・具体例・根拠リンクを含む）。ページが存在しない場合は初回実行時に新規作成し、「要確認」「承認済み」の2セクション見出しを用意する。配置場所（親ページ）は実装時に確認する。既に同一文言のルールが「要確認」または「承認済み」セクションに存在する場合は追記しない（完全一致チェック）。

承認フローはユーザーが手動で「要確認」から「承認済み」へ該当項目を移動することで行う（ステータス列は持たず、ページ内の位置そのものが承認状態を表す）。

### 差分再実行のための状態管理

ニュースリリースDBに新規プロパティ「用語抽出済み」（チェックボックス型）を追加する。このプロパティが存在しない場合はスクリプトが初回実行時にNotion APIでデータベーススキーマに追加する（本番DBのスキーマ変更を伴う）。各実行ではこのチェックが未設定のreleaseのみを対象とし、処理完了後にONにする。Claude APIエラーでバッチ全体がスキップされた場合はONにせず、次回実行で自然に再処理対象になるようにする。

### CLIインターフェース

`prtimes_to_notion.py`と同じ流儀で以下のオプションを用意する。

- `--dry-run`: Notionへの書き込み（用語登録・表記ルール追記・チェックボックス更新）を一切行わずプレビューする
- `--max N`: 処理するrelease件数の上限（0=全件）
- `--batch-size N`: Claude APIに渡す1バッチあたりのrelease件数（デフォルト10）

### エラー処理

- 本文取得失敗: 該当releaseをスキップしログ出力、処理は継続
- Claude APIエラー: バッチ単位で1回リトライ、失敗時はそのバッチをスキップしログ出力（該当releaseの「用語抽出済み」はONにしない）
- Notion書き込みエラー（用語登録時）: 該当用語のみスキップしログ出力、バッチの他の処理は継続

### テスト方針

Claude API・Notionクライアントをモックし、以下を検証する（これまでの修正と同方式のスモークテスト）。

- 用語の重複排除ロジック（既存pending/approvedと一致するものは登録しない）
- Claude応答のJSON解析の頑健性（余分なテキストが混在しても配列部分を抽出できる）
- バッチ分割ロジック（指定件数ごとに正しく分割される）
- 表記ルール統合パスへの入力構築（蓄積したルール候補が正しく渡される）
- 「用語抽出済み」チェックボックスの自動追加ロジック（プロパティが存在しない場合のみ追加を試みる）

## 既知の制約

- 今回のコーパス（PR Times日本語原文）にはKR/EN原文が存在しないため、抽出される用語のKR表記は常に空欄になる。EN表記は本文中にラテン文字表記がある場合のみ抽出される。
- 表記ルールの精度はClaude任せの推定であり、固定の判定基準は存在しない（既存の用語集の運用方針と同様、必ず人間が「要確認」→「承認済み」の移動を通じて確認する）。
- 用語・表記ルールの重複排除はいずれも完全一致チェックである。「KRAFTON」と「(株)KRAFTON」のような表記の異なる同一語句は別エントリとして両方pending登録される可能性がある。これは既存の用語集承認フロー（人間が「要確認」を見て判断する）で吸収する前提とし、今回は厳密な名寄せロジックは作らない。
